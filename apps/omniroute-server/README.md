# OmniRoute Gateway Server

## Purpose

The deployable, standalone implementation of "OmniRoute" described in
`docs/architecture/OMNIROUTE_INTEGRATION.md` — "the default AI Gateway for the platform... every
AI request generated by Wise Content Factory should flow through OmniRoute." Until this package
existed, `packages/infrastructure/src/omniroute.js` was only a reference implementation used
in-process (tests, `examples/`, the browser demo); nothing implemented the hosted HTTP endpoint
that `apps/gateway/OmniRouteProxy.gs`'s `callOmniRouteHttp_` calls when `OMNIROUTE_ENDPOINT` is
configured. This is that endpoint.

## Architecture

```
apps/gateway (Google Apps Script)  --POST-->  apps/omniroute-server (this package)
        |  OMNIROUTE_ENDPOINT +                        |
        |  OMNIROUTE_API_KEY                            v
        |                                    packages/infrastructure/src/omniroute.js
        |                                    (cache -> rank providers -> execute w/ failover
        |                                     -> validate -> cache -> observe)
        |                                                |
        v                                                v
  Browser (apps/web)                          packages/providers/* (Anthropic, OpenAI-compatible,
                                                mock local fallback)
```

`server.js` is a thin HTTP wrapper: it registers real provider adapters from environment
variables, validates and authenticates incoming requests, and forwards everything else to the
same `OmniRoute` class used everywhere else in this repo — no parallel routing/caching/failover
logic was written here, per `CLAUDE.md`'s "never duplicate business logic."

Like every `packages/*` module, this has zero runtime dependencies — plain `node:http`, no
framework. It imports `packages/*` via relative paths (the same pattern `apps/web/index.html`'s
import map uses) rather than through npm workspaces, so it runs standalone with nothing but Node.

## Configuration

All configuration is environment variables — nothing is hard-coded, and no capability is ever
unavailable just because this file wasn't edited (`CLAUDE.md`: "never hard-code provider-specific
behavior").

| Variable | Effect |
| --- | --- |
| `PORT` | Port to listen on (default `8787`). |
| `OMNIROUTE_API_KEY` | Shared secret required as `Authorization: Bearer <key>`. **Unset = unauthenticated dev mode** — the server logs a warning and accepts every request. Always set this before exposing the service beyond localhost. |
| `ANTHROPIC_API_KEY`, `ANTHROPIC_MODEL` | Registers the Anthropic provider (`packages/providers/src/anthropicProvider.js`). |
| `GEMINI_API_KEY`, `GEMINI_MODEL` | Registers the Gemini provider (`packages/providers/src/geminiProvider.js`) — Google's own request format, not OpenAI-compatible. |
| `OPENAI_API_KEY`, `OPENAI_BASE_URL`, `OPENAI_MODEL` | Registers an OpenAI-compatible provider. |
| `DEEPSEEK_API_KEY`, `DEEPSEEK_BASE_URL`, `DEEPSEEK_MODEL` | Registers DeepSeek via the same OpenAI-compatible adapter. |
| `OLLAMA_BASE_URL`, `OLLAMA_MODEL` | Registers a free local-tier provider (e.g. `http://localhost:11434/v1`) — no key needed. |

The local mock text/media providers (`packages/providers/src/mockProvider.js`,
`mockMediaProvider.js`) are **always** registered regardless of configuration, so the service
never has zero providers for a capability — matching OMNIROUTE_INTEGRATION.md's Local-First
Policy (cache → local → free → low-cost → premium) and "never block generation."

## Examples

Run it:

```bash
ANTHROPIC_API_KEY=sk-ant-... OMNIROUTE_API_KEY=change-me npm run omniroute:serve
```

Call it directly (this is exactly what `OmniRouteProxy.gs`'s `callOmniRouteHttp_` sends):

```bash
curl -s http://localhost:8787/health

curl -s http://localhost:8787/ \
  -H "authorization: Bearer change-me" \
  -H "content-type: application/json" \
  -d '{"id":"cap_1","capability":"generate_text","input":{"prompt":"Say hi"},"brandId":null,"projectId":null,"qualityLevel":"balanced","maxCostUsd":null,"preferredProvider":null,"allowCache":true}'
```

Point the gateway at a deployed instance: set `OMNIROUTE_ENDPOINT` (this server's URL) and
`OMNIROUTE_API_KEY` (same shared secret) in the Apps Script project's Script Properties — see
`apps/gateway/README.md`.

See `apps/omniroute-server/test/server.test.mjs` for the full request/response contract, including
auth, validation, and failure-path tests exercised against a real listening `http.Server`.

## Failure Modes

- **No provider supports the requested capability**: `OmniRoute.request` throws; this returns
  `502` with the error message, never a crash — matches "Never terminate an entire workflow
  because one provider is unavailable" (the *caller*, i.e. the gateway/Agent, is expected to
  fail over or proceed without that capability, not this server).
- **Invalid capability request body**: `400`, with the specific validation errors from
  `validateCapabilityRequest`.
- **Malformed JSON body**: `400`.
- **Body over 1MB**: `413` — capability requests are small text/JSON; this is not a media upload
  endpoint.
- **Missing/wrong Bearer token** (when `OMNIROUTE_API_KEY` is set): `401`.
- **Unhandled exception anywhere in the handler**: caught at the top level and returned as `500`
  rather than crashing the process or hanging the connection.
- **No `OMNIROUTE_API_KEY` configured**: the server still runs (useful for local development) but
  logs a warning at startup, since this means anyone who can reach the port can spend your AI
  budget.

## Future Extension Notes

- `search_knowledge` (one of `packages/core/src/schemas/capabilityRequest.js`'s `CAPABILITIES`) is
  not yet bridged to the MCP Manager / NotebookLM MCP integration (`packages/infrastructure/src/mcp/`)
  here — today only `packages/agents/src/agents/researchAgent.js` calls `mcpManager` directly.
  `OMNIROUTE_INTEGRATION.md`'s "MCP Integration" section calls for OmniRoute to expose MCP
  capabilities through the same unified capability endpoint; wiring a `search_knowledge` capability
  to `mcpManager.callTool('knowledge_retrieval', 'search', ...)` here (with cache-through via
  `CacheEngine`, per "Cache Knowledge Results") is the natural next step, deferred to keep this PR
  self-contained.
- No token-level prompt compression yet (`OMNIROUTE_INTEGRATION.md`'s "Compression" /
  "Token Optimization" sections) — providers currently send full prompts as-is.
- No per-brand/per-project routing policy overrides yet (`OMNIROUTE_INTEGRATION.md`'s "Brand
  Policies" / "Project Policies") — `ProviderRouter` already supports `preferredProvider` and
  `maxCostUsd` per request, which is the primitive that policy layer would compile down to.
